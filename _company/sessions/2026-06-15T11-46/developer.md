# 📝 데이터 신뢰성 기반 검증 체크리스트 (Data Reliability Verification Checklist)

**목표:** 향후 모든 기능 추가 시, 시스템의 데이터 무결성을 최우선으로 보장한다.
**원칙:** **신뢰성(Reliability) > 효율성(Efficiency) > 자동화(Automation)**

## 1단계: 설계 단계 (Design Phase) — 신뢰도의 기반 마련

새로운 기능이나 데이터 모델을 도입하기 전에 다음 질문에 대해 명확하게 정의해야 합니다.

1.  **데이터 계층 및 중요도 정의:**
    *   추가되는 모든 데이터 요소($D_{new}$)가 기존의 **Level 1 (Critical)**, **Level 2 (Important)**, **Level 3 (Informational)** 중 어느 계층에 속하는지 명확히 분류했는가?
    *   이 데이터가 최종 KPI(수익 측정)에 직접적인 영향을 미치는가? (Yes/No 명시)

2.  **신뢰도 목표 설정:**
    *   새로운 데이터 스트림($D_{new}$)에 대해 최소 허용 신뢰도 점수 $R_{target}$을 정의했는가? ($R_{target} \ge 0.9$ 권장)

3.  **일관성 규칙 정의 (Consistency Rules):**
    *   이 데이터는 어떤 다른 시스템의 데이터와 반드시 일치해야 하는가? (예: 재고 데이터는 매출 집계와 항상 $T+1$ 시점에 일치해야 함.)
    *   데이터 간의 모순이 발생했을 때의 **롤백/보정(Rollback/Correction) 로직**을 설계했는가?

## 2단계: 구현 단계 (Implementation Phase) — 흐름 및 인터페이스 검증

실제 코드를 작성할 때, 데이터의 이동 경로와 외부 인터페이스를 엄격하게 관리합니다.

1.  **데이터 흐름 무결성 (`src/data_flow.ts` 연관):**
    *   모든 데이터 변환($D_{in} \rightarrow D_{out}$) 단계에서 **손실(Loss) 또는 왜곡(Distortion)**이 발생하지 않음을 보장하는가? (특히, 집계 및 합산 로직에 대한 검증은 필수다.)
    *   데이터 흐름 내 모든 변환 함수는 명시적인 입력/출력 제약 조건(Constraints)을 가지고 있는가?

2.  **API 인터페이스 무결성 (`src/api_interface.ts` 연관):**
    *   외부로 노출되는 모든 API 엔드포인트($E$)에 대해, 요청된 데이터가 **1단계에서 정의한 신뢰도 기준을 충족하는지** 검증하는 가드(Guard) 로직이 구현되었는가? (외부 입력에 대한 강력한 유효성 검사 필수.)
    *   API 응답은 일관된 형식과 신뢰도를 보장하는가?

3.  **시크릿 관리:**
    *   하드코딩된 값(API 키, 임계값 등)이 시스템의 데이터 무결성에 미치는 잠재적 위험을 평가하고, 반드시 환경 변수(`process.env`) 또는 안전한 설정 파일에서 로드하도록 설계했는가?

## 3단계: 검증 단계 (Verification Phase) — 자동화된 자기 검증 루프

코드가 작성된 후, 사람이 아닌 시스템이 데이터 무결성을 자동으로 검증하도록 합니다.

1.  **단위 테스트 확장 (`lint_test` 연관):**
    *   기존의 기능적 테스트 외에 **데이터 무결성 테스트(Integrity Tests)**를 추가했는가? (예: 입력값과 출력값 간의 보정 계수($\lambda$) 검증, 트랜잭션 완료 후 데이터 일치 여부 확인.)
    *   `lint_test` 실행 시, 단순히 코드 구문 오류뿐만 아니라 **데이터 흐름의 논리적 무결성**을 점검하도록 테스트 케이스를 확장했는가?

2.  **종단점 검증 (End-to-End Verification):**
    *   핵심 비즈니스 플로우(예: 주문 생성 $\rightarrow$ 데이터 흐름 $\rightarrow$ API 호출 $\rightarrow$ DB 저장) 전체에 대해 **데이터 신뢰도 보고서**를 자동으로 생성하는 프로세스를 구축했는가?

3.  **알림 및 복구 루프:**
    *   테스트 실패 시, 에러 메시지가 **'코드 오류'**인지 **'데이터 무결성 오류'**인지를 명확히 구분하도록 시스템을 설계했는가? (실패 시 자동 수정/재시도 메커니즘 포함)
